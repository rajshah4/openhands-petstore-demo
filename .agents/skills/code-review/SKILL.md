# Code Review Skill

Use this skill when a pull request is opened, synchronized, reopened, or marked ready for review.

## Goals

1. Review the PR diff for correctness, API behavior, maintainability, and test coverage.
2. Focus on the Petstore app's Java controllers, generated model usage, Maven build, and CI workflows.
3. Run targeted verification when useful.
4. Leave GitHub review feedback or PR comments with clear, actionable findings.

## Repository-specific guidance

- Existing build command: `mvn --no-transfer-progress -B install --file pom.xml`
- Fast verification: `mvn --no-transfer-progress -B test`
- Existing CI already covers Maven builds, dependency review, release flows, and CodeQL.
- There are currently few or no unit tests under `src/test/`, so missing regression coverage is especially important.

## Review checklist

- Does the change alter API responses or status codes in a surprising way?
- Does it preserve existing Petstore data behavior in `PetController`, `OrderController`, or `UserController`?
- Are null cases and not-found cases handled consistently?
- Are workflow files or scripts minimal and safe?
- If code changes are substantial, are there tests or a strong reason why tests are absent?

## Output expectations

- Leave concise GitHub feedback.
- Prioritize real bugs, contract breaks, risky dependency updates, missing tests, and incorrect workflow permissions.
- Do not approve your own PR.
- If the change looks good, leave a short positive summary comment instead of forcing nits.
- Add a brief disclosure note that the comment was generated by OpenHands on behalf of the repository operator.
